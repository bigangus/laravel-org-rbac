# Security

## 报告漏洞

若你发现安全相关问题，请通过仓库的 Issue/私信（按仓库维护者公开渠道）负责任地披露；请勿在公开 Issue 中直接贴出可利用细节。

## 租户解析与信任边界

本包通过 `TenantResolver` 按配置顺序解析「当前租户」，可能来源包括：

- **路由参数**（如 `{tenant}`）：由你的路由定义，通常视为应用内可信标识。
- **HTTP Header**（默认 `X-Tenant-ID`）：**来自客户端**，可被任意构造。仅当请求已经过认证且你在应用层校验「该用户有权访问该租户」时，才可把 Header 解析结果当作当前租户；否则应忽略或拒绝。
- **子域名**：若启用，需结合你的 DNS/网关策略评估是否可被伪造。
- **已认证用户的 `tenant_id` 列**：通常表示用户主租户，仍建议在敏感操作前做成员关系或策略校验。

**不要将「仅解析到的租户 id」等同于授权结果**；功能权限请继续使用 `hasOrgRbacPermission*`、Laravel **Policy/Gate**，并在控制器或中间件中显式校验租户成员关系。

## 超级管理员

当用户模型的 `is_super_admin` 为 true 时，包内权限检查会视为拥有全部权限。请确保：

- 该字段只能由可信后台流程写入；
- 应用层 Policy 若需对超管例外，应显式记录业务规则，避免与包内行为混淆。

## 缓存

权限有效结果可缓存（TTL、前缀、可选 tag）。在共享 Redis 等多租户共用缓存前缀的场景下，请使用应用级缓存键前缀隔离；启用 tag 时需使用支持 tag 的缓存驱动。

自 **0.6** 起，生产环境默认要求 **默认缓存驱动为 Redis**（`Illuminate\Cache\RedisStore`），以便权限缓存失效路径（tag flush 或 SCAN）行为可预期。

## 审计日志（建议）

本包通过 **`OrgRbacLog`** 输出结构化日志（`org_rbac => true`），覆盖例如：租户解析失败、租户树重绑（含 `old_path` / `new_path`）、按 tag 或 SCAN 清理权限缓存等。**这不等同于业务审计**：你仍应在应用中记录至少：

- **主体**：用户 id、客户端 IP（注意隐私法下的合法性基础）、请求 id / trace id。  
- **客体**：租户 id、资源类型与 id、权限名、角色变更。  
- **结果**：允许 / 拒绝（含原因码）、异常。  
- **时间**：UTC 或可统一审计时区。

建议在应用层对 **超管操作、租户切换、角色/权限变更、鉴权失败** 单独落库或写入 **只追加（append-only）** 的审计流（如 SIEM、对象存储 + 不可变保留策略）。

## 合规与外部审计（文档级说明）

以下内容用于协助 **SOC 2、ISO 27001、等保** 等体系下的 **控制设计**，**不是**法律意见或认证承诺：

- **数据驻留**：租户与 RBAC 数据存于你的数据库与 Redis；请在你的隐私政策/DPA 中声明跨境与分包（如云 Redis）。  
- **日志留存**：按法规设定保留期与删除流程；包内日志请纳入你们日志基线。  
- **渗透测试 / 代码审计**：上线前建议对 **集成本包的应用** 做黑盒与依赖项扫描；开源依赖见 `composer.lock`。  
- **责任边界**：**多租户隔离、Header 信任、超管治理** 由应用策略与运维保证；本包提供机制与说明，**不替代**贵司安全开发生命周期（SDL）。

若需 **正式合规证据**，请委托持证第三方或贵司法务/合规团队出具评估报告。
